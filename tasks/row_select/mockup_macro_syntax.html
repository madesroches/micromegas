Row Selection — Macro Syntax Options

Three syntax alternatives for referencing selected row columns in downstream cells.

Syntax Option 1: $cell.selected.column recommended

Introduces a .selected accessor on the cell name. Reads naturally as "the selected row's column from this cell". New regex pattern, doesn't conflict with existing $cell[N].col syntax.

SQL Example
-- upstream cell "processes" has a selected row
SELECT time, level, target, msg
FROM view_instance('log_entries', '$processes.selected.process_id')
WHERE time >= '$from'
ORDER BY time DESC
Markdown Example
Process: $processes.selected.exe
Started at: $processes.selected.start_time
User: $processes.selected.username
Pros
  • Reads naturally: "processes selected process_id"
  • Self-documenting — .selected is explicit
  • Easy regex: $name.selected.col
  • Doesn't conflict with existing $cell[N].col
  • Consistent with $variable.column dotted style
Cons
  • Longer than bracket syntax
  • "selected" is a reserved word (can't name a column "selected")

Syntax Option 2: $cell[selected].column

Uses the existing bracket accessor but with the keyword selected instead of a numeric index. Maximum consistency with $cell[0].column syntax.

SQL Example
-- same pattern as $cell[0].col but dynamic
SELECT time, level, target, msg
FROM view_instance('log_entries', '$processes[selected].process_id')
WHERE time >= '$from'
ORDER BY time DESC
Pros
  • Maximally consistent with $cell[0].col
  • Bracket position = row selector (number or keyword)
  • Extends existing regex naturally
Cons
  • Less readable in long expressions
  • Bracket syntax feels more "programmatic"
  • Harder to distinguish from numeric refs at a glance

Syntax Option 3: $selected.cell.column

Prefix-style with $selected as a namespace. Immediately signals that this value depends on user interaction.

SQL Example
-- $selected namespace makes dependency obvious
SELECT time, level, target, msg
FROM view_instance('log_entries', '$selected.processes.process_id')
WHERE time >= '$from'
ORDER BY time DESC
Pros
  • Clear visual signal of interactive dependency
  • All selected refs have same prefix — easy to search
  • Won't conflict with any cell named "selected"
Cons
  • Inconsistent with existing $cell[N].col pattern
  • Three dots: $selected.cell.column is deep nesting
  • "selected" could collide with a variable named "selected"
  • Ambiguous: is "processes" a cell name or column?

Comparison

Use Case Option 1 ($cell.selected.col) Option 2 ($cell[selected].col) Option 3 ($selected.cell.col)
Access process_id $processes.selected.process_id $processes[selected].process_id $selected.processes.process_id
In SQL string '$p.selected.id' '$p[selected].id' '$selected.p.id'
Co-exists with $cell[0].col No conflict Same pattern family No conflict
Co-exists with $variable.col Disambiguated by "selected" keyword Disambiguated by brackets Could conflict with $selected variable
Regex complexity New 3-part pattern Extend existing bracket pattern New 3-part pattern

Recommendation: Option 1 ($cell.selected.column) balances readability with consistency. It reads naturally in both SQL and markdown, doesn't conflict with existing patterns, and the .selected keyword clearly communicates the interactive nature of the reference. Option 2 is the best alternative if maximal consistency with $cell[N].col is preferred.
