Expand description

Same as split_logs, but folds extra_hash_input into the block_id hash alongside the pre-backfill ResourceLogs bytes. Used by the webhook path (handler::ingest_webhook) to fold in the full incoming HTTP header set: the synthetic ResourceLogs only carries the 3 recognized X-Micromegas-* headers as resource attrs, so without this, any other header (a delivery-id, a signature, an event-type header) would have zero influence on block_id — two deliveries with the same body but different unrecognized headers would collide and dedup as if they were retries of each other. Passing &[] (what split_logs does) reproduces the OTLP-only behavior exactly.
