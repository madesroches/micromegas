1//! Scalar math UDFs for 1D remapping and related helpers.
2//!
3//! # API conventions
4//!
5//! These conventions apply to every math UDF in this crate. Once external
6//! SQL exists in the wild they are effectively frozen — any new math UDF
7//! must honour them, and behaviour-changing variants must adopt a distinct
8//! name.
9//!
10//! - **No clamping.** `lerp(a, b, t)` with `t` outside `[0, 1]` extrapolates;
11//!   `unlerp(a, b, x)` with `x` outside `[a, b]` returns a value outside
12//!   `[0, 1]`. Callers who want clamping wrap with
13//!   `LEAST(GREATEST(t, 0.0), 1.0)`.
14//! - **IEEE-754 propagation, not errors.** `NaN`/`±∞` inputs propagate via
15//!   float math. Degenerate inputs (e.g. `unlerp(a, a, x)`) produce `NaN`
16//!   or `±Inf` rather than per-row errors. Callers who want a fallback
17//!   wrap with the existing `nanvl(...)` built-in.
18//! - **Nulls propagate.** If any input is `NULL`, the row's result is
19//!   `NULL`. Matches every other scalar UDF in this crate.
20//! - **Float64 only.** All signatures use `Signature::exact(vec![Float64; N], …)`.
21//!   DataFusion's implicit numeric coercion lets callers write integer
22//!   literals (`lerp(0, 1, 0.5)`) without explicit casts.
23//!
24//! Future-extension naming reservations (recorded so the API stays coherent
25//! as it grows):
26//! - `smoothstep(edge0, edge1, x) -> Float64` — HLSL/GLSL-style smooth
27//!   Hermite interpolation. Composes with `lerp` the same way `unlerp` does.
28//! - `map_range(x, a, b, c, d) -> Float64` — one-shot remap. Currently
29//!   superseded by `lerp(c, d, unlerp(a, b, x))`; name reserved.
30//! - `saturate(x) -> Float64` — HLSL-style `clamp(x, 0, 1)`. Sibling to
31//!   wrap around `unlerp` results when callers want `[0, 1]`-clamped output.
