1//! Binning UDFs for snapping continuous coordinates to discrete grids.
2//!
3//! # API conventions
4//!
5//! These conventions apply to every binning UDF in this crate. Once external
6//! SQL exists in the wild they are effectively frozen — any new binning UDF
7//! must honour them, and behaviour-changing variants must adopt a distinct
8//! name.
9//!
10//! - **Centered on zero.** `bin_center(0, cs) = 0`. The bin containing
11//!   `coord` spans the half-open interval `[c - cs/2, c + cs/2)` where
12//!   `c` is the returned center.
13//! - **Half-open intervals.** A point that lands exactly on a bin edge
14//!   (`coord = c + cs/2`) belongs to the *next* bin. This matches the standard
15//!   `FLOOR` convention used by the `histogram` module's bucketing.
16//! - **Pathological inputs are not validated.** `cell_size <= 0` and
17//!   `NaN`/`±∞` inputs are not rejected. Float math propagates `NaN`/`Inf`
18//!   naturally; a zero or negative `cell_size` produces undefined results.
19//!   Documented as a precondition.
20//!
21//! Future-extension naming reservations (recorded so the API stays coherent
22//! as it grows):
23//! - `bin_index(coord, cs) -> Int64` — the raw `floor(...)` result, for
24//!   sparse-key use cases.
25//! - Vectorized 2D/N-D variants are deliberately not offered; the 1D form
26//!   composes trivially (`bin_center(x, cs), bin_center(y, cs)`).
